# Чек-лист тестирования REST API: /api/v2/orders

**Основание для проверок:**
- Google REST API Guide (AIP)
- Microsoft Graph REST API Guidelines (MS Graph)
- Стандарты HTTP (RFC 7231)

**Метки в чек-листе:**
- `[AIP]` — требование из Google API Design Guide
- `[MS Graph]` — требование из Microsoft Graph REST API Guidelines
- `[HTTP]` — требование стандарта HTTP

---

## Структура ответов сервера

- **Успех (`2xx`)**: Ответ **ОБЯЗАН** соответствовать контракту `OrderReadModel` (JSON).
- **Ошибка валидации (`400`)**: Ответ **ОБЯЗАН** содержать тело с детализацией ошибки согласно структуре `error` с полями `code`, `message`, `innererror` `[MS Graph]`.
- **Ошибка прав (`403`)** и **Не найдено (`404`)**: Допустимо пустое тело или структура `error`.

## 1. Позитивные и бизнес-сценарии

### `GET /api/v2/orders` (List)

| № | Сценарий | Ожидаемый статус | Комментарий |
|---|----------|------------------|-------------|
| 1.1 | Запрос списка при наличии заказов | `200 OK` | Тело: коллекция в объекте `value` (OData) или массив |
| 1.2 | Запрос списка при отсутствии заказов | `200 OK` | Тело: пустой массив `[]` (или `{"value": []}`) |
| 1.3 | Проверка логического удаления | `200 OK` | Записи с `isDeleted: true` отсутствуют в списке |
| 1.4 | `[MS Graph]` Пагинация через `@odata.nextLink` | `200 OK` | Ответ содержит `@odata.nextLink`, если данных больше лимита |
| 1.5 | `[MS Graph]` Клиентская пагинация (`$top` / `$skip`) | `200 OK` | `?$top=5&$skip=10` возвращает ожидаемое подмножество |
| 1.6 | `[MS Graph]` Проекция свойств (`$select`) | `200 OK` | `?$select=id,displayName` возвращает только указанные поля |
| 1.7 | `[MS Graph]` Сортировка (`$orderby`) | `200 OK` | `?$orderby=createdDateTime desc` сортирует корректно |
| 1.8 | `[MS Graph]` Фильтрация (`$filter`) | `200 OK` | `?$filter=status eq 'ACTIVE'` возвращает отфильтрованные записи |
| 1.9 | `[MS Graph]` Подсчет (`$count`) | `200 OK` | `?$count=true` добавляет `@odata.count` в ответ |

### `GET /api/v2/orders/{id}` (Get)

| № | Сценарий | Ожидаемый статус | Комментарий |
|---|----------|------------------|-------------|
| 2.1 | Запрос существующей активной записи | `200 OK` | Тело: `OrderReadModel` |
| 2.2 | Запрос с несуществующим в природе UUID | `404 Not Found` | ID никогда не создавался |
| 2.3 | Запрос логически удаленной записи | `410 Gone` | Запись существует, но `isDeleted: true` |
| 2.4 | `[MS Graph]` Раскрытие связей (`$expand`) | `200 OK` | `?$expand=items` возвращает заказ с вложенными товарами |

### `POST /api/v2/orders` (Create)

| № | Сценарий | Ожидаемый статус | Комментарий |
|---|----------|------------------|-------------|
| 3.1 | Успешное создание (ID генерирует сервер) | `201 Created` | Тело: `OrderReadModel`. Заголовок `Location` обязателен |
| 3.2 | Успешное создание с клиентским ID (`POST /.../{id}`) | `201 Created` | Создан новый ресурс |
| 3.3 | Повторный вызов `POST /.../{id}` с тем же ID | `409 Conflict` | Попытка создать дубликат |
| 3.4 | `[MS Graph]` Создание через `PATCH` (Upsert) | `201 Created` | `PATCH` на несуществующий ID. Заголовок `Location` присутствует |

### `PATCH /api/v2/orders/{id}` (Update)

> `[MS Graph]` Microsoft Graph требует использовать `PATCH` вместо `PUT` для обновлений.

| № | Сценарий | Ожидаемый статус | Комментарий |
|---|----------|------------------|-------------|
| 4.1 | Частичное обновление существующей записи (с телом) | `200 OK` | Тело: обновленная `OrderReadModel` |
| 4.2 | Частичное обновление (без возврата тела) | `204 No Content` | Тело ответа пустое |
| 4.3 | Попытка обновить логически удаленную запись | `410 Gone` | Запрещено менять удаленное |

## 2. Кросс-функциональные и негативные сценарии

### 2.1. Валидация входных данных (Ожидаемый статус: `400 Bad Request`)

`[MS Graph]` Код ошибки обязан быть `badRequest`; специфика — в `innererror.code`.

| № | Сценарий | Ожидаемый `innererror.code` (пример) |
|---|----------|--------------------------------------|
| V.1 | Некорректный JSON в теле запроса | `invalidJson` |
| V.2 | Неверный тип данных (строка вместо числа) | `typeMismatch` |
| V.3 | Отсутствует обязательное поле (`customerId`) | `requiredFieldMissing` |
| V.4 | Выход за границы значений (имя 10 000 символов) | `fieldTooLong` |
| V.5 | Отрицательное количество товара | `invalidValue` |
| V.6 | Неверный формат UUID в URL (`/orders/123`) | `invalidUrlFormat` |
| V.7 | `[MS Graph]` Неизвестный параметр `$select` или `$filter` | `unknownProperty` |

### 2.2. Аутентификация и авторизация

| № | Сценарий | Ожидаемый статус | Комментарий |
|---|----------|------------------|-------------|
| A.1 | Запрос без токена (`Authorization` отсутствует) | `401 Unauthorized` | |
| A.2 | `[MS Graph]` Недостаточно прав (ReadOnly → `POST`) | `403 Forbidden` | |
| A.3 | `[MS Graph]` Сокрытие данных (чужая запись) | `404 Not Found` | Допустимо вместо `403` по privacy-соображениям |
| A.4 | `[HTTP]` Method Not Allowed (`POST` на ID без поддержки) | `405 Method Not Allowed` | |

### 2.3. Отказы инфраструктуры

| № | Сценарий | Ожидаемый статус | Комментарий |
|---|----------|------------------|-------------|
| I.1 | Таймаут соединения с БД | `504 Gateway Timeout` | |
| I.2 | БД недоступна (connection refused) | `503 Service Unavailable` | По возможности — заголовок `Retry-After` |
| I.3 | `[MS Graph]` Превышение клиентской квоты | `429 Too Many Requests` | **Обязателен** заголовок `Retry-After` |
| I.4 | Неожиданное исключение (NPE, баг) | `500 Internal Server Error` | Без стектрейса в теле ответа |

## 3. Чек-лист соответствия стандартам

### 3.1. Формат ошибок `[MS Graph]` (Критично)

| № | Проверка | Критерий приемки |
|---|----------|------------------|
| E.1 | Структура ответа | Корень JSON содержит объект `"error"` |
| E.2 | Поле `code` | `error.code` существует всегда. Значение в camelCase (`badRequest`, `notFound`) |
| E.3 | Поле `message` | Человекочитаемое описание ошибки |
| E.4 | Поле `innererror` | При детализации `error.innererror.code` содержит специфичный код (`requiredFieldMissing`) |

### 3.2. Правила нейминга полей `[MS Graph]`

| № | Проверка | Критерий приемки |
|---|----------|------------------|
| N.1 | Стиль имен | Все свойства в JSON используют **lower camelCase** (`displayName`). Без `snake_case` |
| N.2 | Булевы свойства | Начинаются с `is` или `has`: `isDeleted`, `isEnabled`, `hasChildren` |
| N.3 | Дата и время | Суффиксы: `Date`, `Time`, `DateTime`. Пример: `createdDateTime`, `dueDate` |
| N.4 | ID ресурса | Строка (`String`), ключ в JSON — `id` (строчные), не `ID` |
| N.5 | Коллекции | Имена массивов во множественном числе: `items`, `addresses` |
| N.6 | Запрещенные суффиксы | Нет `Collection`, `Response`, `Request` в именах свойств |
| N.7 | Запрет брендов | Имена не содержат кодовых названий продуктов (`teamsOrder`) |

### 3.3. Заголовки и контракт данных

| № | Проверка | Критерий приемки |
|---|----------|------------------|
| H.1 | Content-Type | `200`/`201`/`400`: `application/json`. Ошибки — всегда JSON |
| H.2 | Формат дат | ISO 8601 / RFC 3339 (строка). Timestamp (long) запрещен |
| H.3 | Перечисления (enum) | Строковые значения (`"PENDING"`), не числа |
| H.4 | OData-аннотации | Коллекции содержат `@odata.context` (желательно) |
| H.5 | Deprecation | Если версия устарела — заголовок `Deprecation` присутствует |

### 3.4. Версионирование и обратная совместимость `[MS Graph]`

| № | Проверка | Критерий приемки |
|---|----------|------------------|
| V.1 | Изоляция версий | `/v2/orders` не затрагивает логику `/v1/orders` |
| V.2 | Старые поля | Переименованное поле в v2 — в v1 остается с пометкой `isDeprecated: true` |
| V.3 | Жесткий контракт | Добавление нового обязательного поля без смены версии — **Breaking Change** (запрещено) |
