### Промт для LLM

**Роль:**
Ты — ведущий инженер по качеству (Senior QA) в стартапе по логистике. Твоя задача — составить подробный, структурированный чек-лист для тестирования MVP формы «Рассчитать стоимость перевозки» для сервиса «УЛ Лайт».

**Контекст задачи:**
Мы проверяем клиентскую часть (фронтенд) и её взаимодействие с бэкендом. Нам важно не только проверить ввод данных, но и корректность обработки ответов от сервера.

**Описание формы и полей:**
1.  **Город отправления** (`originCity`): Поле с автокомплитом. Значения подтягиваются из базы данных. Пользователь вводит текст, система предлагает варианты. Нельзя ввести город, которого нет в БД.
2.  **Город назначения** (`destinationCity`): Аналогичное поле с автокомплитом.
3.  **Вес груза** (`weight`): Числовое поле. Единица измерения — кг. Ограничение: максимум 20 000 кг.
4.  **Объем груза** (`volume`): Числовое поле. Единица измерения — м³. Ограничение: максимум 82 м³.
5.  **Дата погрузки** (`loadingDate`): Компонент «Датапикер». Пользователь не должен иметь технической возможности выбрать прошедшую дату (блокировка на уровне календаря).
6.  **Кнопка «Рассчитать стоимость»**: Отправляет POST-запрос с данными формы. Успешный ответ (200 OK) содержит стоимость в рублях. Если маршрут не найден, API возвращает ошибку валидации (422 или 400) с телом ошибки, которую нужно показать пользователю. На время запроса кнопка должна блокироваться (состояние загрузки).

**Требования к чек-листу:**
Сформируй ответ в виде таблицы Markdown со следующими колонками:
*   **ID**: Уникальный номер теста (формат F-XXX).
*   **Тип проверки**: (Позитивная / Негативная / Граничные значения / UI/UX).
*   **Название проверки**: Краткое и емкое описание сути теста.
*   **Шаги**: Конкретные действия или предусловия.
*   **Ожидаемый результат**: Что должно произойти после выполнения шагов.

**Обязательные области для покрытия (проследи, чтобы были учтены все):**
1.  **Позитивные сценарии:**
    *   Успешный расчет с валидными данными.
    *   Проверка формата возвращаемых данных (рубли, копейки).
    *   Сброс формы до начального состояния.
2.  **Поля «Город отправления» и «Город назначения» (Автокомплит):**
    *   Поиск по подстроке (например, по 2-3 буквам).
    *   Выбор значения из выпадающего списка (клик/Enter).
    *   Очистка выбранного значения (крестик).
    *   Проверка регистрозависимости.
    *   Ввод несуществующего города и потеря фокуса.
3.  **Поля «Вес» и «Объем» (Числовые):**
    *   Граничные значения: `0`, `1`, `20000` / `82` (максимум), `20001` / `83` (превышение максимума).
    *   Ввод отрицательных чисел (например, `-5`).
    *   Ввод букв и спецсимволов (`е`, `@`, `+`).
    *   Ввод дробных чисел (с точкой и запятой, например `10.5` и `10,5`).
    *   Ввод пробелов или пустого значения.
4.  **Поле «Дата погрузки» (Датапикер):**
    *   Функциональная блокировка прошедших чисел (визуально и кликабельно).
    *   Ручной ввод прошедшей даты с клавиатуры (если поле это позволяет).
    *   Выбор сегодняшней даты.
    *   Выбор будущей даты.
5.  **Интеграция с API (Кнопка «Рассчитать стоимость»):**
    *   Поведение при отправке формы с незаполненными обязательными полями (фронт-валидация до отправки запроса).
    *   Поведение кнопки во время ожидания ответа (дизейбл, спиннер/текст "Расчет...").
    *   Отображение ошибки "Маршрут не найден" от сервера (например, города есть, но между ними нет рейсов).
    *   Симуляция сетевой ошибки (Timeout, 500 Internal Server Error): выводится ли понятное сообщение без падения интерфейса.
    *   Быстрый двойной клик по кнопке (отправляется ли два запроса?).

**Формат вывода:**
Выведи готовую Markdown таблицу без лишних комментариев до и после неё. Сгруппируй проверки по блокам.

### Почему этот промт хорош (как сеньор сеньору):
1.  **Фиксация роли:** Слова «ведущий инженер по качеству» и «стартап» настраивают модель на практичный лад без «воды».
2.  **Атрибуты полей:** Я указал названия в коде (`originCity`). Это сразу намекает модели, что нужно проверять `name` в запросе, если она будет писать автотесты.
3.  **Специфика автокомплита:** LLM часто забывают про очистку поля (крестик) и потерю фокуса без выбора значения.
4.  **Тонкость датапикера:** Я акцентировал внимание на «ручном вводе с клавиатуры» — это классический баг, когда календарь не дает выбрать прошедшее число, но скрипт или прямое редактирование поля — позволяет.
5.  **Двойной клик:** Обязательный пункт для проверки идемпотентности кнопки, про который джуны часто забывают.
6.  **Защита от сетевых ошибок:** В MVP часто ломаются на показе сырых ответов 500-й ошибки прямо в интерфейсе, я попросил проверить это явно.
