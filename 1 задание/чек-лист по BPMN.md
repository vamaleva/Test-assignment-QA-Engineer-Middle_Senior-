## Блок 1. Happy Path (Сквозной успешный прогон)

| ID | Приоритет | Шаг (Action) | Ожидаемый результат |
| :---: | :---: | :--- | :--- |
| HP-01 | 🔴 | Инициатор создает новый ЭД, заполняет все обязательные поля. | ЭД создан, присвоен уникальный номер. |
| HP-02 | 🔴 | Инициатор подписывает ЭД (УКЭП) и отправляет на согласование. | Статус: «На согласовании у Инвестора». Владелец сменяется на Инвестора. |
| HP-03 | 🔴 | Инвестор успешно согласует ЭД. | Статус: «На подписании у Поставщика». |
| HP-04 | 🔴 | Поставщик согласует (ромб = «Да») и подписывает УКЭП. | Документ возвращается Инициатору. |
| HP-05 | 🔴 | Инициатор утверждает ЭД и подписывает со своей стороны. | Обе подписи валидны. |
| HP-06 | 🔴 | Финальная передача в Архив. | ЭД помещен в раздел «Постоянное хранение». Редактирование запрещено. |
| HP-07 | 🟡 | Проверка PDF-визуализации в Архиве. | В штампе отображаются все три подписи (Инициатора, Инвестора, Поставщика) без наложений. |

## Блок 2. Передача прав и блокировка действий (Владельцы дорожек)

| ID | Приоритет | Проверка | Ожидаемый результат |
| :---: | :---: | :--- | :--- |
| BL-01 | 🔴 | ЭД у Инвестора. Инициатор пытается редактировать или подписать документ. | Кнопки/API методы «Редактировать», «Подписать» недоступны. |
| BL-02 | 🔴 | ЭД у Инвестора. Поставщик пытается получить доступ к документу. | Документ не виден в интерфейсе Поставщика до завершения шага «Утвердить». |
| BL-03 | 🟡 | Инициатор меняет свои данные в справочнике, пока ЭД на подписании у Поставщика. | Данные внутри уже отправленного ЭД остаются неизменными (Point-in-time snapshot). |
| BL-04 | 🟡 | Два пользователя (сессии) одновременно пытаются выполнить «Направить на согласование». | Система пропускает только первую транзакцию, вторая уходит в «Lock timeout» или ошибку. |

## Блок 3. Первый цикл отката Поставщика (Ветвление «Согласовано? Нет»)

| ID | Приоритет | Проверка | Ожидаемый результат |
| :---: | :---: | :--- | :--- |
| PS-01 | 🔴 | Поставщик отклоняет документ с комментарием «Неверная цена». | ЭД возвращается к Инициатору. Статус: «Доработка». Комментарий от Поставщика виден. |
| PS-02 | 🔴 | Инициатор исправляет цену и повторно отправляет. | Создается новая версия ЭД (ревизия). Старая версия доступна только для чтения. |
| PS-03 | 🔴 | Провести цикл «Отказ -> Доработка -> Отправка» 3 раза подряд. | Процесс не зависает. Каждый возврат корректно меняет владельца. Ошибки 5xx отсутствуют. |
| PS-04 | 🟡 | Поставщик отклоняет документ, но не заполняет причину отказа (валидация полей). | Система блокирует отправку, подсвечивая обязательное поле «Причина отказа». |

## Блок 4. Второй цикл отката (Ветвление «Отказ?» — Разрыв сделки)

| ID | Приоритет | Проверка | Ожидаемый результат |
| :---: | :---: | :--- | :--- |
| OT-01 | 🔴 | Инициатор получает правки, не согласен с ними. Выбирает «Отказаться от подписания». | Статус ЭД: «Расторгнут» / «Отменен». Дальнейшие действия заблокированы. |
| OT-02 | 🔴 | Проверить, что при отказе не отправляется команда на подписание УКЭП. | В логах Оператора ЭДО отсутствует исходящий запрос на финальную подпись. |
| OT-03 | 🟡 | Процесс завершен отказом. Все стороны (включая Инвестора, если настроено) получают уведомление. | В центре уведомлений есть сообщение «Сделка расторгнута». |
| OT-04 | 🟡 | Инициатор соглашается с правками (Нет = Согласие). | Процесс переходит на этап «Передать на подписание Поставщику» (возврат в Happy Path). |

## Блок 6. Электронный архив (Финальная стадия)

| ID | Приоритет | Проверка | Ожидаемый результат |
| :---: | :---: | :--- | :--- |
| AR-01 | 🔴 | Документ помещен в Архив. Попытка редактирования любой из сторон (включая админа). | Полный запрет на изменение контента (иммутабельность). Допускается только скачивание/печать. |
| AR-02 | 🟡 | Инициатор пытается отправить в Архив ЭД, где не хватает подписи Поставщика. | Система блокирует передачу. Ошибка: «Не все подписи собраны». |
| AR-03 | 🟡 | Поиск документа в Архиве по реквизитам (номер, дата, контрагент). | Документ находится за время не более 3 секунд (SLA). |
| AR-04 | 🟡 | Автоматическая передача: Прервать связь с сервером Архива на моменте «Передать ЭД». | Документ остается в статусе «Отправка в архив». При восстановлении связи система автоматически ретраит отправку. |

## Блок 7. Интеграция и Оператор ЭДО (Теневая дорожка)

| ID | Приоритет | Проверка | Ожидаемый результат |
| :---: | :---: | :--- | :--- |
| INT-01 | 🔴 | Отправка документа Поставщику. Имитация тайм-аута (502/504 Gateway). | ЭД не уходит в статус «Ошибка чтения». Статус: «Ошибка отправки». Доступна кнопка «Повторить отправку». |
| INT-02 | 🔴 | Идемпотентность: Инициатор дважды быстро нажимает «Направить на подписание». | Создается только один исходящий пакет у Оператора ЭДО. Второй запрос игнорируется (HTTP 409 Conflict). |
| INT-03 | 🔴 | Проверка валидности цепочки УКЭП (Математическая проверка). | ПО Крипто-шлюза (например, КриптоПро DSS) подтверждает валидность всех подписей в цепочке после финального шага. |
| INT-04 | 🟡 | Попытка отправить документ Контрагенту с аннулированным сертификатом ЭП. | Ошибка: «Сертификат получателя недействителен». Отправка блокируется до обновления сертификата. |

## Блок 8. Ролевая модель и безопасность (RBAC)

| ID | Приоритет | Проверка | Ожидаемый результат |
| :---: | :---: | :--- | :--- |
| SEC-01 | 🔴 | Пользователь с ролью «Поставщик» вставляет прямую ссылку на страницу «Согласование Инвестором». | Ошибка 403 (Forbidden) или редирект на главную. |

## Блок 9. Регресс и обработка исключений

| ID | Проверка | Ожидаемый результат |
| :---: | :--- | :--- |
| REG-01 | ЭД на стадии «Создан». Инициатор удаляет документ (если разрешено). | Удаление возможно только до первой отправки. После отправки кнопка «Удалить» скрывается (замена на «Аннулировать», если актуально). |
| REG-02 | Поставщик не отвечает 7 дней. | Документ должен отображаться в гриде с предупреждением о просрочке (SLA Warning). Регламентная отмена не должна срабатывать без предупреждения. |
| REG-03 | Проверка кросс-браузерности работы плагина КриптоПро на шаге подписания. | Подписание доступно в браузерах, поддерживающих расширение (Chromium-based, Firefox ESR). |
