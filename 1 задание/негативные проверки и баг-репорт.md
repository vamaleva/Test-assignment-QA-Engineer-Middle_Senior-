### Блок «Процесс согласования (BPMN)»

| № | Проверка | Ожидаемое поведение |
|---|---|---|
| 1 | **Проверка:** Инициатор создал ЭД и отправил Инвестору. В этот момент (до согласования Инвестором) Инициатор через прямой API-запрос пытается вызвать метод `/api/docs/sendToArchive`. | **Ожидаемое поведение:** Запрос отклонен (HTTP 409 Conflict или 403 Forbidden). Ответ API: *«Операция запрещена. Документ находится на этапе согласования, передача в архив невозможна»*. |
| 2 | **Проверка:** Поставщик на этапе подписания пытается прикрепить сертификат, срок действия которого истек вчера (просроченная УКЭП). | **Ожидаемое поведение:** Система вызывает метод проверки сертификата (OCSP/CRL), фиксирует, что сертификат недействителен, и выбрасывает ошибку на UI. Подпись не формируется. Сообщение: *«Ошибка подписания: сертификат пользователя недействителен или отозван»*. |
| 3 | **Проверка (Регресс):** Инициатор исправляет документ после возврата, но на этот раз вносит сумму, превышающую лимит, установленный для конкретного Инвестора, и снова отправляет ему. | **Ожидаемое поведение:** При попытке Инвестора согласовать такой документ система жестко блокирует действие с ошибкой *«Превышен лимит полномочий. Требуется эскалация»*. Статус ЭД остается неизменным. |

## Часть 2. Баг-репорты

---

### Баг-репорт №1 (Бизнес-логика + Безопасность)

**Заголовок:** ЭД в статусе «На подписании у Поставщика» не блокирует редактирование данных Инициатором через API сохранения черновика

**Серьёзность:** 🔴 Критическая (Блокирует целостность данных и процесс подписания)

**Окружение:**  
- Тестовый стенд: `tst-edo`  
- Сборка: `5.01.12`  
- Пользователь: `Initiator_Ivanov`

**Шаги воспроизведения:**
1. Инициатор создает и отправляет ЭД на согласование (статус «На согласовании у Инвестора»).
2. Инвестор выполняет вход в систему, согласует документ.
3. Инициатор отправляет документ на подписание Поставщику. Статус документа в UI меняется на «На подписании у Поставщика».
4. Инициатор в соседней вкладке открывает DevTools (или использует Postman/SoapUI).
5. Инициатор авторизуется под своей УЗ, берет ID текущего документа.
6. Инициатор направляет PUT/PATCH запрос к API `api/documents/{id}/draft` с телом, содержащим измененную (уменьшенную) цену контракта: `{ "price": 1 }`.
7. API возвращает статус `200 OK`.
8. Инициатор обновляет страницу просмотра документа. Данные изменились.

**Фактический результат:**
Система разрешила модификацию данных контента документа в обход бизнес-процесса. Документ, ожидающий подписания контрагентом, был изменен задним числом без увеличения версии и без возврата на этап «Доработка».

**Ожидаемый результат:**
Согласно чек-листу (кейс BL-01), на этапе «Подписание у Поставщика» владелец дорожки — Поставщик. Инициатор не может редактировать содержимое. API должен проверять текущий статус документа и возвращать ошибку **HTTP 422 Unprocessable Entity** с текстом: *«Редактирование запрещено. Документ находится на этапе обработки контрагентом»*.
Данные должны остаться неизменными (Point-in-time snapshot, кейс BL-03).

### Баг-репорт №2 (Форматный контроль ЭДПГ)

**Заголовок:** Система успешно принимает файл КНД 1110429 с `Функция=1`, но с заполненным полем `ПричРастрж` (нарушение Schematron)

**Серьёзность:** 🟡 Значительная (Нарушение логики НСИ, риск некорректной интерпретации документа контрагентом)

**Окружение:**  
- Модуль приема файлов: `XML-Validator`  
- Версия: `1.8.3`

**Шаги воспроизведения:**
1. Сформировать XML-документ КНД 1110429 (Согласование соглашения).
2. В разделе атрибутов указать `Функция` = `"1"` (Согласование).
3. Нарушая п. 4.4.2 и 4.4.3 чек-листа, добавить в файл атрибуты:
   `<ПричРастрж>Ошибка</ПричРастрж>`
   `<ДатаРастрж>11.06.2026</ДатаРастрж>`
4. При этом блок `СвСглс` заполнен корректно.
5. Загрузить файл на проверку.

**Фактический результат:**
Файл проходит входной контроль. XML-документ уходит в обработку. На UI отображается статус «Загружен». Система никак не сигнализирует о противоречивых данных. Создается риск, что на стороне получателя такой файл будет трактован неоднозначно (как согласование с одновременным указанием причины расторжения).

**Ожидаемый результат:**
Валидатор должен отклонить файл с ошибкой контрольного соотношения. Проверка должна срабатывать до этапа загрузки в реестр документов.
Текст ошибки должен ссылаться на п. 4.4.2: *«Нарушение логической целостности Функция=1: поля ПричРастрж и ДатаРастрж должны отсутствовать, так как блок СвСглс присутствует»*.
Статус файла — «Ошибка ФЛК».
